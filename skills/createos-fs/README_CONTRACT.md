## Purpose

This contract defines **what the backend README must contain** and **what it must not**.

It exists to ensure that:

* a user can run the backend without guidance
* setup steps are explicit and reproducible
* assumptions are documented
* the backend is operable once dependencies are available

The README is an **operator guide**, not a design document.

---

## Core Principle

> **If someone cannot run the backend using the README alone, the README is incomplete.**

---

## Tone & Style Rules

The README must be:

* concise
* direct
* technical
* free of fluff

Avoid:

* long explanations
* tutorials
* justification of design decisions

---

## Forbidden Content

The README must NOT:

* Explain internal architecture (contracts handle this)
* Include code walkthroughs
* Include deployment guides unless explicitly requested
* Contain TODOs or vague instructions

---

## Success Criteria

This contract is satisfied if:

* A developer can set up and run the backend in one sitting
* No prior context is required
* All required conon is discoverable
* Deferred features are clearly called out

---

## Failure Conditions

This contract is violated if:

* Steps are missing or ambiguous
* Environment variables are undocumented
* Database setup requires guessing
* Swagger location is unclear
* Placeholders are undocumented

---

## Summary

This contract ensures the README is:

* practical
* complete
* operator-focused
* trustworthy

> **The README is the handshake between the backend and its user.**
